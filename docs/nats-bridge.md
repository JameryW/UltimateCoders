# NATS Bridge Architecture

> How the gRPC TaskService, Python Orchestrator, and Dashboard communicate
> through NATS JetStream for task lifecycle coordination.

---

## Overview

The NATS bridge connects three independent processes:

1. **Rust gRPC server** — serves TUI TaskService RPCs (`submitTask`, `getTask`,
   `listTasks`, `watchTask`, `pauseTask`, `resumeTask`)
2. **Python NATS consumer** (`nats_worker`) — subscribes to task submissions,
   runs the real Orchestrator (LLM decomposition + Worker execution), publishes
   state updates back
3. **Dashboard** (FastAPI) — optionally publishes tasks through NATS instead of
   calling the Orchestrator directly; subscribes to task events for SSE streaming

When NATS is unavailable, both the gRPC server and Dashboard gracefully degrade
to local/in-process processing (no cross-process coordination, but still functional).

## NATS Subjects and Payloads

### `uc.task.submit` — New task submission

Published by: gRPC server, Dashboard

Consumed by: `nats_worker`

```json
{
  "task_id": "uuid-string",
  "description": "Fix the login bug",
  "project_id": "proj-1"
}
```

The `task_id` is generated by the publisher. The consumer receives it and
passes it to `Orchestrator.submit_task()`, which may reassign the ID during
decomposition (the original ID is used only for tracking the NATS correlation).

### `uc.task.update` — Task state update

Published by: `nats_worker` (via `NatsPublisher`)

Consumed by: gRPC server (updates in-memory `TaskStore`)

```json
{
  "task_id": "uuid-string",
  "status": "InProgress",
  "subtasks": [
    {
      "subtask_id": "st-uuid",
      "status": "Assigned",
      "assigned_worker": "worker-1",
      "result": null
    }
  ],
  "result": null
}
```

Status strings use CamelCase matching the Rust `TaskStatus` enum: `Created`,
`Planning`, `InProgress`, `Completed`, `Failed`, `Paused`.

Subtask status strings: `Pending`, `Assigned`, `InProgress`, `Completed`,
`Failed`, `Conflicted`.

### `uc.task.event` — Real-time event

Published by: `nats_worker` (via `NatsPublisher`)

Consumed by: gRPC server (pushes into `TaskStore.event_log` for `WatchTask`
streaming), Dashboard (merges into SSE stream)

```json
{
  "type": "subtask_assigned",
  "task_id": "uuid-string",
  "subtask_id": "st-uuid",
  "data": {
    "worker_id": "worker-1"
  }
}
```

Event types recognized by the Rust gRPC server:

| Event type | Rust `AgentEventType` variant |
|---|---|
| `subtask_assigned` | `SubtaskAssigned` |
| `subtask_started` | `SubtaskStarted` |
| `tool_call` | `ToolInvoked` |
| `tool_result` | `ToolResult` |
| `file_modified` | `FileModified` |
| `subtask_completed` | `SubtaskCompleted` |
| `subtask_failed` | `SubtaskFailed` |

The Dashboard also publishes `task_pause` and `task_resume` event types on this
subject for control commands routed through NATS.

### `uc.heartbeat` — Consumer heartbeat

Published by: `nats_worker` (every 30 seconds)

Consumed by: gRPC server (heartbeat monitor)

```json
{
  "consumer_id": "consumer-uuid",
  "timestamp": "2026-06-16T12:00:00Z"
}
```

If no heartbeat is received within the configured timeout (default 10 minutes),
the gRPC server marks all `InProgress`/`Planning` tasks as `Failed`.

## Process Lifecycle

### gRPC server (Rust)

```
GrpcServer::new(engine)                  -- no NATS, local decomposition
GrpcServer::with_nats(engine, nats_url)  -- NATS connected, subscriber + monitor
```

When NATS is connected:

1. `submit_task` publishes to `uc.task.submit`, creates a `Planning`-status
   task in `TaskStore` (no subtasks yet)
2. Background subscriber listens on `uc.task.update` and `uc.task.event`,
   updating `TaskStore` accordingly
3. Heartbeat monitor checks `uc.heartbeat`; marks stale tasks `Failed`

When NATS publish fails, falls back to local (newline-split) decomposition,
removing the `Planning`-status placeholder task first.

### Python NATS consumer (`nats_worker`)

```
python -m ultimate_coders.nats_worker
```

Environment variables:

| Variable | Default | Purpose |
|---|---|---|
| `UC_NATS_URL` | `nats://localhost:4222` | NATS server URL |
| `UC_SANDBOX_MODE` | (empty) | `"subprocess"` for sandbox execution |
| `UC_PROJECT_PATH` | current directory | Project path for sandbox |

Lifecycle:

1. Connects to NATS (with retry, exponential backoff up to 5 attempts)
2. Initializes `Engine` (local mode) + `Orchestrator` + `Worker`
3. Subscribes to `uc.task.submit`
4. For each submission: calls `Orchestrator.submit_task()`, then
   `assign_subtask()` and `execute_subtask()` for each ready subtask
5. Publishes `uc.task.update` and `uc.task.event` after each state change
6. Publishes `uc.heartbeat` every 30 seconds

### Dashboard (FastAPI)

```
DashboardApp(orchestrator, nats_publisher=..., nats_client=...)
DashboardApp.from_env(orchestrator)  -- auto-connect from UC_NATS_URL
```

When NATS publisher is configured:

- `submit_task` endpoint publishes to `uc.task.submit` via
  `NatsPublisher.publish_submit()`, returns `Planning` status immediately
- `pause_task` / `resume_task` publish control events via
  `NatsPublisher.publish_event()`

When NATS client is configured (for event subscription):

- Subscribes to `uc.task.event`, pushes received events into an
  `asyncio.Queue` that the SSE stream merges with local
  `TaskEventEmitter` events

When NATS is not available, both paths fall back to direct Orchestrator calls
(legacy mode, unchanged behavior).

## Data Flow: TUI Submit to Task Completion

```
TUI                     gRPC server            NATS            nats_worker        Orchestrator
 |                          |                    |                 |                  |
 |-- submitTask(desc) ----> |                    |                 |                  |
 |                          |-- uc.task.submit ->|                 |                  |
 |                          |  (Planning task)   |-- subscribe --> |                  |
 |                          |                    |                 |-- submit_task -->|
 |                          |                    |                 |                  |-- decompose(LLM)
 |                          |                    |                 |                  |-- assign_subtask
 |                          |                    |<-- uc.task.update (InProgress + subtasks)
 |                          |<-- subscriber -----|                 |                  |
 |                          |  TaskStore updated  |<-- uc.task.event (subtask_assigned)
 |<-- WatchTask stream ---- |                    |                 |                  |-- execute_subtask
 |                          |                    |<-- uc.task.update (subtask completed)
 |                          |<-- subscriber -----|                 |                  |
 |                          |                    |<-- uc.task.update (Completed)
 |                          |<-- subscriber -----|                 |                  |
 |<-- WatchTask: Completed--|                    |                 |                  |
```

## Data Flow: Dashboard Submit (NATS mode)

```
Dashboard              NATS            nats_worker        Orchestrator
 |                      |                 |                  |
 |-- POST /submit ----> |                 |                  |
 |  (NatsPublisher)     |-- uc.task.submit->|                 |
 |                      |                 |-- submit_task -->|
 |<-- 200 Planning -----|                 |                  |-- decompose
 |                      |<-- uc.task.update|                  |
 |  (SSE subscribes)    |<-- uc.task.event |                  |
 |<-- SSE: task_event --|                 |                  |
```

## Graceful Degradation

| Scenario | gRPC server behavior | Dashboard behavior |
|---|---|---|
| NATS unavailable at startup | No subscriber; local decomposition only | No NATS publish/subscribe; direct Orchestrator calls |
| NATS publish fails mid-request | Remove `Planning` placeholder, re-submit with local decomposition | Fall back to direct Orchestrator call |
| NATS consumer crashes | Heartbeat timeout; tasks marked `Failed` | SSE continues with local `TaskEventEmitter` events only |
| NATS reconnects | No auto-reconnect (requires server restart) | No auto-reconnect (requires dashboard restart) |

## Docker Compose Configuration

The `nats-worker` service in `docker-compose.yml` runs the NATS consumer as
an independent container:

```yaml
nats-worker:
  build: .
  depends_on:
    pd: { condition: service_healthy }
    tikv: { condition: service_healthy }
    qdrant: { condition: service_healthy }
    postgres: { condition: service_healthy }
    nats: { condition: service_healthy }
  environment:
    UC_NATS_URL: "nats://nats:4222"
    UC_TIKV_PD_ENDPOINTS: "pd:2379"
    UC_QDRANT_URL: "http://qdrant:6334"
    UC_PG_URL: "postgresql://ultimate_coders:ultimate_coders@postgres:5432/ultimate_coders"
    UC_SANDBOX_MODE: "subprocess"
    UC_PROJECT_PATH: "/workspace"
  command: ["python", "-m", "ultimate_coders.nats_worker"]
```

The `orchestrator` service continues to run with its embedded Dashboard. When
both are running, task submissions from the Dashboard go through NATS (if
`UC_NATS_URL` is set in the orchestrator's environment), ensuring the gRPC
TaskStore stays in sync with TUI consumers.

## Known Limitations

1. **No auto-reconnect** -- If NATS disconnects mid-stream, the gRPC subscriber
   and Dashboard subscription exit. A server restart is required to re-establish
   the connection.

2. **Task ID may change** -- When the NATS consumer receives a submission with
   a publisher-generated `task_id`, the Orchestrator may create a different ID
   during `submit_task()`. The gRPC TaskStore tracks the original ID (from the
   `Planning` placeholder), but NATS updates reference the Orchestrator's ID.
   This is a known mismatch that will be resolved when the Orchestrator accepts
   an external task ID.

3. **Subtask events lack `task_id`** -- The Rust `AgentEventType` variants for
   subtask-level events (`SubtaskAssigned`, `SubtaskStarted`, etc.) do not
   carry a `task_id` field. The TUI `WatchTask` with a specific `task_id`
   may miss subtask events. Current workaround: TUI watches all tasks (empty
   `task_id`).

4. **Heartbeat timeout is coarse** -- The heartbeat monitor checks every 30
   seconds with a 10-minute default timeout. Fine-grained timeout (e.g., per-task
   deadline) is not yet implemented.

5. **No JetStream persistence** -- The current setup uses core NATS (at-most-once
   delivery). If the gRPC subscriber is disconnected when an update is published,
   that update is lost. JetStream persistence would provide at-least-once delivery
   but requires additional configuration (stream/consumer setup).
