# Research: buf + Connect-ES gRPC-Web TypeScript Client Setup

- **Query**: How to set up buf for generating gRPC-Web TypeScript clients using Connect-ES
- **Scope**: Mixed (internal project structure + external library documentation)
- **Date**: 2026-06-17

## Findings

### Critical Discovery: Connect v2 Eliminates Separate Connect Plugin

In Connect v2 (current, released 2024-2025), `@connectrpc/protoc-gen-connect-es` has been **removed entirely**. Service descriptors are now generated by `@bufbuild/protoc-gen-es` alone. There is no longer a separate `*_connect.ts` file -- everything goes into `*_pb.ts` files. Service types like `ElizaService` are exported directly from the protobuf-generated code.

This means:
- NO `protoc-gen-connect-es` plugin is needed in `buf.gen.yaml`
- NO `@connectrpc/protoc-gen-connect-es` npm package is needed
- A single `protoc-gen-es` plugin generates both message types AND service descriptors
- Import paths change from `*_connect` to `*_pb`

### buf.gen.yaml Configuration

The correct `buf.gen.yaml` for this project (v2 format):

```yaml
# buf.gen.yaml
# Learn more: https://buf.build/docs/configuration/v2/buf-gen-yaml
version: v2
clean: true
inputs:
  - directory: ../crates/uc-grpc/proto
plugins:
  - local: protoc-gen-es
    out: src/grpc
    opt:
      - target=ts
      - import_extension=js   # needed for Node16/bundler module resolution with ESM
```

**Key points about this config:**

1. **`version: v2`** -- Required for the current buf.gen.yaml format
2. **`clean: true`** -- Deletes the output directory before regenerating, preventing stale files
3. **`inputs`** -- Uses `directory` input to point at the proto source directory. This is the correct approach for local proto files (as opposed to BSR modules like `buf.build/connectrpc/eliza`)
4. **`local: protoc-gen-es`** -- Uses the locally-installed plugin from `node_modules/.bin/`
5. **`target=ts`** -- Generates TypeScript output (alternative: `target=js` for JS+d.ts, `target=dts` for declarations only)
6. **`import_extension=js`** -- Adds `.js` extensions to import paths, needed when using ESM with Node16/bundler module resolution in TypeScript. Without this, TypeScript's `--moduleResolution bundler` or `node16` may fail on extensionless imports
7. **`include_imports`** -- NOT needed here because `engine.proto` has no imports from other proto files. If it did (e.g., google/protobuf), set `include_imports: true`

**Alternative: Using remote plugin instead of local**

```yaml
version: v2
clean: true
inputs:
  - directory: ../crates/uc-grpc/proto
plugins:
  - remote: buf.build/bufbuild/es:v2.12.0
    out: src/grpc
    opt:
      - target=ts
      - import_extension=js
```

The remote plugin avoids needing `@bufbuild/protoc-gen-es` installed locally but requires BSR access.

### buf CLI Installation

**Recommended: via npm/pnpm (project-local)**

The `@bufbuild/buf` npm package (v1.71.0) contains the full buf CLI binary. This is the recommended approach for JavaScript/TypeScript projects because:

1. It pins the buf version in `package.json` alongside other dev dependencies
2. It works with `pnpm` without requiring a global install
3. It's the approach used in the official Connect-ES examples
4. No Homebrew or manual binary installation needed

```bash
# Install as devDependency
pnpm add -D @bufbuild/buf

# Run via pnpm
pnpm buf generate

# Or via npx (if installed)
npx buf generate
```

**Other installation methods (not recommended for this project):**

| Method | Command | Notes |
|---|---|---|
| Homebrew | `brew install bufbuild/buf/buf` | macOS-only, version may lag |
| Docker | `docker run --rm -v $(pwd):/workspace bufbuild/buf generate` | Heavy, CI-focused |
| Binary install | `curl -sSL "https://github.com/bufbuild/buf/releases/latest/download/buf-Darwin-arm64" -o /usr/local/bin/buf` | Manual updates |
| npm global | `npm install -g @bufbuild/buf` | Works but version not pinned to project |

**npm script for code generation:**

```json
{
  "scripts": {
    "generate": "buf generate"
  }
}
```

### Server-Streaming Support (WatchTask)

Both browser transports support server-streaming RPCs:

1. **`createGrpcWebTransport`** -- Uses the gRPC-Web protocol. Requires a gRPC-Web proxy (e.g., Envoy) or a server that natively supports gRPC-Web (e.g., `tonic-web` in Rust)

2. **`createConnectTransport`** -- Uses the Connect protocol. Server-streaming works via fetch API with readable streams. The Connect protocol is HTTP-based and simpler, but the Rust server needs to serve the Connect protocol (via a Connect-Go/Connect-Rust library or a reverse proxy that translates)

**Streaming usage pattern (from generated code):**

```typescript
import { createClient } from "@connectrpc/connect";
import { createGrpcWebTransport } from "@connectrpc/connect-web";
import { TaskService } from "./grpc/engine_pb.js";

const transport = createGrpcWebTransport({
  baseUrl: "http://localhost:8080",  // gRPC-Web proxy or tonic-web endpoint
});

const client = createClient(TaskService, transport);

// Server-streaming call (WatchTask)
for await (const event of client.watchTask({ taskId: "abc123" })) {
  console.log(event.type, event.taskId, event.data);
}
```

**Important: gRPC-Web proxy requirement**

The Rust tonic server speaks native gRPC (HTTP/2 + protobuf). Browsers cannot speak native gRPC. You need one of:

1. **tonic-web** (Rust crate) -- Enables tonic to serve gRPC-Web directly. This is the simplest path for our project. Add `tonic-web` to the Rust server and enable it as a layer.

2. **Envoy proxy** -- Adds a gRPC-Web filter. More infrastructure but production-proven.

3. **Connect protocol** -- If the server adds Connect protocol support, `createConnectTransport` works directly over HTTP/1.1 with fetch. However, no mature Rust Connect server library exists yet.

**Recommendation for this project**: Use `tonic-web` crate on the Rust side + `createGrpcWebTransport` on the frontend. This is the simplest path since we already have a tonic server.

### Generated Code Structure

With the configuration above, buf will generate:

```
dashboard/src/grpc/
  engine_pb.ts          # All message types + service descriptors for engine.proto
```

The generated file exports:
- TypeScript type aliases for each message (e.g., `type TaskEvent = Message<"ultimate_coders.TaskEvent"> & {...}`)
- Schema objects for each message (e.g., `TaskEventSchema: GenMessage<TaskEvent>`)
- Service descriptor objects (e.g., `TaskService: GenService<{...}>`)
- File descriptor (e.g., `file_engine: GenFile`)

**Usage of generated types:**

```typescript
import { create } from "@bufbuild/protobuf";
import { TaskService, WatchTaskRequestSchema, type TaskEvent } from "./grpc/engine_pb.js";

// Create a request message
const request = create(WatchTaskRequestSchema, { taskId: "abc123" });

// Use the service descriptor with createClient
const client = createClient(TaskService, transport);
```

### Vite Proxy Configuration

For development, the Dashboard's Vite dev server needs to proxy gRPC-Web requests to the Rust server. Update `vite.config.ts`:

```typescript
export default defineConfig({
  server: {
    port: 5173,
    proxy: {
      // Existing REST/SSE proxy
      "/dashboard/api": {
        target: "http://localhost:8080",
        changeOrigin: true,
      },
      // gRPC-Web proxy (tonic-web serves at same port under /ultimate_coders.TaskService/...)
      // gRPC-Web uses POST to paths like /ultimate_coders.EngineService/Search
      "^/ultimate_coders\\..*/": {
        target: "http://localhost:8080",
        changeOrigin: true,
        // gRPC-Web requires these headers
        configure: (proxy) => {
          proxy.on("proxyReq", (proxyReq) => {
            // Vite http-proxy may strip content-type; preserve it
          });
        },
      },
    },
  },
});
```

Alternatively, configure the transport with the full server URL to bypass the proxy:

```typescript
const transport = createGrpcWebTransport({
  baseUrl: "http://localhost:8080",  // Direct to Rust server
});
```

## Caveats / Not Found

1. **tonic-web not yet integrated** -- The Rust gRPC server (`crates/uc-grpc`) does not currently use `tonic-web`. This crate MUST be added for the browser to communicate via gRPC-Web. Without it, `createGrpcWebTransport` will fail to connect.

2. **Connect protocol vs gRPC-Web protocol** -- The Connect protocol (`createConnectTransport`) is simpler and works over standard HTTP/1.1, but no Rust Connect server implementation exists. The gRPC-Web protocol (`createGrpcWebTransport`) requires tonic-web on the server side but is well-supported.

3. **CORS** -- If the Rust server runs on a different port than the Dashboard, CORS headers must be configured on the gRPC server side. tonic-web may need explicit CORS configuration.

4. **Vite proxy for gRPC-Web** -- The standard Vite HTTP proxy may not handle gRPC-Web streaming correctly (it uses `http-proxy` which may buffer responses). Direct connection or a custom proxy may be needed for server-streaming in development.

5. **ESM import extensions** -- The `import_extension=js` option in buf.gen.yaml is needed for ESM projects. The current dashboard uses `"type": "module"` in package.json, and TypeScript with bundler module resolution. Testing is needed to confirm whether this option is required.

6. **No buf.yaml needed** -- For simple single-file proto generation, only `buf.gen.yaml` is needed. A `buf.yaml` (for linting, breaking change detection, BSR publishing) is optional.

7. **The `inputs` field with `directory`** -- This uses a path relative to the `buf.gen.yaml` location. Since `buf.gen.yaml` will be in `dashboard/` and the proto is at `crates/uc-grpc/proto/`, the relative path `../crates/uc-grpc/proto` is correct. However, this path must be verified at generation time -- buf does not follow symlinks by default.
