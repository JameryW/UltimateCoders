### 1. Proto Definitions & Naming Convention

The entire gRPC surface lives in **one file**: `crates/uc-grpc/proto/engine.proto` (proto3, `package ultimate_coders`). Four services are declared (engine.proto:4-53):

- `EngineService` (engine.proto:4-19)
- `TaskService` (engine.proto:23-32)
- `WorkerService` (engine.proto:36-40) — register/heartbeat/deregister
- `DashboardService` (engine.proto:44-53) — ListWorkers + scheduler/CB/events + WatchDashboard stream

**Existing WorkerService messages** (engine.proto:621-653):

```proto
message RegisterWorkerRequest {
    string worker_id = 1;
    repeated string capabilities = 2;
    uint32 max_capacity = 3;    // max concurrent subtasks
    string metadata = 4;        // optional JSON metadata
}
message RegisterWorkerResponse {
    bool success = 1;
    string worker_id = 2;
    optional string error = 3;
}
message WorkerHeartbeatRequest {
    string worker_id = 1;
    uint32 current_load = 2;
}
message WorkerHeartbeatResponse {
    bool accepted = 1;
    optional string error = 2;
}
message DeregisterWorkerRequest {
    string worker_id = 1;
}
message DeregisterWorkerResponse {
    bool success = 1;
    optional string error = 2;
}
```

**Naming/style conventions observed:**

- Field names: `snake_case` (e.g. `worker_id`, `max_capacity`, `current_load`).
- Response wrappers: every mutating/status RPC returns `bool success = 1` (or `bool accepted = 1`) as field **1**, plus `optional string error` (always `optional`, typically field 2 or 3). The proto3 `optional` keyword is used liberally for nullable scalars/strings (e.g. `optional string error`, `optional string local_path`).
- Empty requests use a bare `{}` message (e.g. `ListWorkersRequest {}`, `HealthRequest {}`).
- Enum-like values are passed as `string` (e.g. content_type `"text"/"code"/"diff"`, dispatch_mode `"Local"/"Remote"/"PreferRemote"`) rather than proto enums — the codebase favors string-typed discriminators.
- `uint32` is used for counts/capacities; `int64` for timestamps (epoch millis); `double` for ratios/ages; `map<string,string>` for event payloads.
- Comments are inline `//` above fields explaining semantics (e.g. `// max concurrent subtasks`).

### 2. Where to Add ScaleWorkers — WorkerService (confirmed)

`WorkerService` is the natural home. Evidence:

- `WorkerService` already owns **worker lifecycle mutation**: `RegisterWorker`, `WorkerHeartbeat`, `DeregisterWorker` (engine.proto:36-40). Scaling is a lifecycle mutation.
- The `WorkerService` impl (`worker_service.rs:185-255`) operates directly on `self.worker_registry().write().await` — the in-memory `WorkerRegistry`. A `ScaleWorkers` RPC would do the same.
- `DashboardService` is explicitly a **read-only NATS passthrough** (dashboard_service.rs:1-6): "All DashboardService RPCs forward via NATS request-reply to the Python Orchestrator." Its `list_workers` (dashboard_service.rs:38-93) only *reads* the registry as a fallback. Putting a mutating scale RPC here would violate its read-only contract.
- `WorkerRegistry` already has the primitives a scale operation needs: `register()` (worker_service.rs:66-94), `deregister()` (worker_service.rs:110-117), `workers()` / `available_workers()` (worker_service.rs:120-130). No new registry method is strictly required for a registry-level scale, though a helper could be added.

**Caveat on semantics**: `WorkerRegistry` is an *in-memory registry of connected workers* — it tracks workers that have *self-registered* over gRPC. It does NOT spawn/stop worker processes. The gateway cannot literally start/stop Python worker processes (they are independent deployment units, per CLAUDE.md). So `ScaleWorkers` can only:
  (a) **declaratively adjust a desired target** (requires a process supervisor / docker compose / NATS command to a controller — none exists today), or
  (b) **force-deregister** workers from the registry (drop them from the in-memory map; they stop receiving subtasks).

The registry's `deregister()` (worker_service.rs:110-117) is a pure map removal — it does NOT notify the worker process to shut down. The worker would only notice when its heartbeats start failing or it stops receiving NATS subtask dispatches.

### 3. Request/Response Shape — Proposed Proto

Given the codebase conventions (string-typed action discriminators, `bool success` + `optional string error` wrappers, `uint32` counts), and the constraint that the gateway cannot spawn processes, the cleanest design covers **both** declarative process-scaling intent AND registry-force-deregister in one RPC, using a string `action` field. This mirrors how `uc_worker` already uses `action: "list" | "status"` (worker-bridge.ts:16) and how `dispatch_mode` uses string enums.

Add to `WorkerService` (engine.proto:36-40):

```proto
service WorkerService {
    rpc RegisterWorker(RegisterWorkerRequest) returns (RegisterWorkerResponse);
    rpc WorkerHeartbeat(WorkerHeartbeatRequest) returns (WorkerHeartbeatResponse);
    rpc DeregisterWorker(DeregisterWorkerRequest) returns (DeregisterWorkerResponse);
    rpc ScaleWorkers(ScaleWorkersRequest) returns (ScaleWorkersResponse);   // NEW
}
```

New messages (append after `DeregisterWorkerResponse`, engine.proto:~653):

```proto
// ── Scale Workers (declarative cluster scaling + registry force-deregister) ──

message ScaleWorkersRequest {
    // Action discriminator (string-typed, per repo convention):
    //   "scale_to"   — set desired worker count to target_count (process scaling).
    //                  Gateway records desired count; actual spawn/stop is delegated
    //                  to an external supervisor (docker compose / NATS controller).
    //                  If no supervisor is wired, returns success=false with an error
    //                  explaining process scaling is unavailable.
    //   "scale_by"   — add (positive delta) or remove (negative delta) N workers.
    //                  delta_count carries the signed change; target_count ignored.
    //   "deregister" — force-remove a specific worker_id from the in-memory
    //                  WorkerRegistry WITHOUT stopping its process. Equivalent to
    //                  DeregisterWorker RPC but exposed as a tool action. Use this
    //                  to evict stale/ghost workers that have crashed without
    //                  graceful deregister.
    string action = 1;

    // For action="scale_to": desired total worker count.
    uint32 target_count = 2;

    // For action="scale_by": signed change (+N to add, -N to remove).
    //   Use int32 to allow negative values (uint32 cannot represent removals).
    int32 delta_count = 3;

    // For action="deregister": specific worker to force-evict from registry.
    string worker_id = 4;

    // Optional capabilities filter for scale_to/scale_by — new workers should
    // advertise these capabilities (passed to the supervisor as a hint).
    repeated string capabilities = 5;

    // Optional max_capacity hint for newly spawned workers.
    uint32 max_capacity = 6;
}

message ScaleWorkersResponse {
    bool success = 1;
    // Current desired worker count after applying the action (for scale_to/scale_by).
    uint32 desired_count = 2;
    // Current actual registered worker count (post-action snapshot).
    uint32 actual_count = 3;
    // Worker IDs affected by the action (e.g. evicted worker_ids for deregister,
    // or newly-requested worker_ids for scale). May be empty for scale_to/scale_by
    // if the supervisor assigns IDs asynchronously.
    repeated string affected_worker_ids = 4;
    optional string error = 5;
}
```

**Design rationale (per repo patterns):**

- `string action` mirrors `uc_worker`'s `action: "list"|"status"` (worker-bridge.ts:16) and `dispatch_mode` string enums — the codebase consistently avoids proto `enum` for action types.
- `bool success = 1` + `optional string error = 5` matches every other mutating response (`RegisterWorkerResponse`, `DeregisterWorkerResponse`, `PauseTaskResponse`).
- `int32 delta_count` (not `uint32`) because removals need negative values — the only signed-int field type available for counts. `uint32` is used elsewhere only for non-negative magnitudes.
- `desired_count` / `actual_count` give the caller a post-action snapshot, consistent with `ListWorkersResponse` returning `total` + `available_count` (engine.proto:427-432).
- `affected_worker_ids` is a `repeated string` like `required_capabilities` (engine.proto:341) — the codebase uses `repeated string` for ID lists everywhere.

**Should deregister be a separate tool action calling the existing DeregisterWorker RPC?**

Two viable paths, both consistent with the codebase:

1. **Unified `ScaleWorkers` with `action="deregister"`** (proposed above) — one RPC, one tool. Pros: single TS bridge method, single `uc_worker` action enum extension, the LLM gets one coherent "worker management" surface. Cons: slightly conflates process-scaling with registry-eviction.
2. **Separate `uc_worker` action `"deregister"` that calls the existing `DeregisterWorker` RPC** — no new proto message needed for the deregister case; only `scale_to`/`scale_by` go in `ScaleWorkers`. Pros: reuses the already-implemented, already-Python-wired `DeregisterWorker` RPC (client.rs:240-255). Cons: the TS side currently has NO `WorkerService` client at all (see §4), so calling `DeregisterWorker` from TS requires adding a `workerClient` to `GrpcBridge` regardless — at which point bundling it into `ScaleWorkers` is marginally more code but one fewer client to wire.

**Recommendation encoded in the proto above**: unify under `ScaleWorkers`. The Rust impl for `action="deregister"` can delegate internally to `registry.deregister(&req.worker_id)` (worker_service.rs:110) — same code path as `DeregisterWorker`, no duplication. The existing `DeregisterWorker` RPC stays for the Python worker's own graceful-shutdown path (nats_worker.py:1221) and is untouched.

### 4. TS Bridge Pattern

**Critical finding**: the TS `GrpcBridge` (grpc-bridge.ts) does NOT create a `WorkerService` client today. It only instantiates (grpc-bridge.ts:106-108, 122-124, 136-138):

```ts
private engineClient: ReturnType<typeof createClient<typeof EngineService>>;
private taskClient: ReturnType<typeof createClient<typeof TaskService>>;
private dashboardClient: ReturnType<typeof createClient<typeof DashboardService>>;
```

The `uc_worker` tool (worker-bridge.ts) is **read-only** — it calls `bridge.listWorkers()` (worker-bridge.ts:33), which routes through `DashboardService.ListWorkers` (grpc-bridge.ts:580-604). There is no TS path to `WorkerService` today.

However, the generated `WorkerService` descriptor IS available in `engine_pb.ts` (engine_pb.ts:2899) — `protoc-gen-es` emits it for every service in the proto. So adding a TS client is purely a wiring task, no codegen gap.

**Pattern a new `scaleWorkers()` method would follow** (mirroring `listWorkers()` at grpc-bridge.ts:580-604 + the `withReconnect` wrapper used by all other RPCs):

```ts
// 1. Add to imports (grpc-bridge.ts:13-38) — WorkerService is already exported by engine_pb.ts
import {
    EngineService, TaskService, DashboardService,
    WorkerService,                        // NEW
    // ... existing Schema imports ...
    ScaleWorkersRequestSchema,            // NEW (generated after proto change + regen)
    ListWorkersRequestSchema,
    // ...
} from "../grpc/engine_pb.js";

// 2. Add a workerClient field + instantiate it (3 places: constructor + reconnect())
export class GrpcBridge {
    private workerClient: ReturnType<typeof createClient<typeof WorkerService>>;   // NEW
    constructor(config?: Partial<BridgeConfig>) {
        // ...
        this.engineClient = createClient(EngineService, this.transport);
        this.taskClient = createClient(TaskService, this.transport);
        this.dashboardClient = createClient(DashboardService, this.transport);
        this.workerClient = createClient(WorkerService, this.transport);          // NEW
    }
    reconnect(): void {
        // ... same 4 clients recreated ...
        this.workerClient = createClient(WorkerService, this.transport);          // NEW
    }

    // 3. New method — follows the withReconnect + create(Schema, {...}) pattern
    async scaleWorkers(params: {
        action: "scale_to" | "scale_by" | "deregister";
        targetCount?: number;
        deltaCount?: number;
        workerId?: string;
        capabilities?: string[];
        maxCapacity?: number;
    }): Promise<{ success: boolean; desiredCount: number; actualCount: number; affectedWorkerIds: string[]; error?: string }> {
        return this.withReconnect(async () => {
            const resp = await this.workerClient.scaleWorkers(
                create(ScaleWorkersRequestSchema, {
                    action: params.action,
                    targetCount: params.targetCount ?? 0,
                    deltaCount: params.deltaCount ?? 0,
                    workerId: params.workerId ?? "",
                    capabilities: params.capabilities ?? [],
                    maxCapacity: params.maxCapacity ?? 0,
                }),
            );
            this.connected = true;
            return {
                success: resp.success,
                desiredCount: resp.desiredCount,
                actualCount: resp.actualCount,
                affectedWorkerIds: [...resp.affectedWorkerIds],
                error: resp.error ?? undefined,
            };
        }, { success: false, desiredCount: 0, actualCount: 0, affectedWorkerIds: [] });
    }
}
```

**Note on `create()` + Schema pattern**: every TS RPC call uses `create(SomeRequestSchema, { ...fields })` from `@bufbuild/protobuf` (grpc-bridge.ts:39, examples at lines 266, 300, 386, 483, 511, etc.). Field names in `create()` are **camelCase** (proto snake_case → TS camelCase via protoc-gen-es), e.g. `worker_id` → `workerId`, `target_count` → `targetCount`, `delta_count` → `deltaCount`. This is visible in the generated types (engine_pb.ts SearchRequest maps `max_results`→`maxResults`, `repo_ids`→`repoIds`).

### 5. Rust Trait Method Signature to Implement

`WorkerService` is implemented via `#[tonic::async_trait]` on `GrpcServer<E>` (worker_service.rs:185-186). Adding `ScaleWorkers` to the proto forces a new trait method. Following the exact signature shape of `deregister_worker` (worker_service.rs:238-241):

```rust
// In crates/uc-grpc/src/worker_service.rs, inside:
// #[tonic::async_trait]
// impl<E: EngineApi + Send + Sync + 'static> WorkerService for GrpcServer<E> {

async fn scale_workers(
    &self,
    request: Request<ScaleWorkersRequest>,
) -> Result<Response<ScaleWorkersResponse>, Status> {
    let req = request.into_inner();
    let mut registry = self.worker_registry().write().await;

    match req.action.as_str() {
        "deregister" => {
            // Delegate to existing registry.deregister() (worker_service.rs:110)
            match registry.deregister(&req.worker_id) {
                Ok(()) => {
                    let actual = registry.workers().len() as u32;
                    Ok(Response::new(ScaleWorkersResponse {
                        success: true,
                        desired_count: actual,   // no desired-count tracking yet
                        actual_count: actual,
                        affected_worker_ids: vec![req.worker_id],
                        error: None,
                    }))
                }
                Err(e) => Ok(Response::new(ScaleWorkersResponse {
                    success: false,
                    desired_count: registry.workers().len() as u32,
                    actual_count: registry.workers().len() as u32,
                    affected_worker_ids: vec![],
                    error: Some(e),
                })),
            }
        }
        "scale_to" | "scale_by" => {
            // Process scaling: gateway cannot spawn processes.
            // Record desired count (would need a new field on GrpcServerInner or
            // a NATS publish to an external supervisor — neither exists today).
            // For now: return success=false with an explanatory error, OR
            // publish a NATS command if a supervisor subject is defined.
            Ok(Response::new(ScaleWorkersResponse {
                success: false,
                desired_count: registry.workers().len() as u32,
                actual_count: registry.workers().len() as u32,
                affected_worker_ids: vec![],
                error: Some(
                    "Process scaling not implemented — gateway cannot spawn/stop \
                     worker processes. Use action=\"deregister\" to evict workers \
                     from the registry, or orchestrate docker compose externally."
                        .to_string(),
                ),
            }))
        }
        other => Ok(Response::new(ScaleWorkersResponse {
            success: false,
            desired_count: 0,
            actual_count: registry.workers().len() as u32,
            affected_worker_ids: vec![],
            error: Some(format!("Unknown action: '{other}' (expected scale_to|scale_by|deregister)")),
        })),
    }
}
```

**Accessor already exists**: `self.worker_registry()` returns `&Arc<RwLock<WorkerRegistry>>` (server.rs:1337-1339). No new server plumbing needed.

**Note on `desired_count` tracking**: `WorkerRegistry` (worker_service.rs:23-181) has no `desired_count` field. If `scale_to`/`scale_by` are to be meaningful, either (a) add a `desired_count: AtomicU32` to `WorkerRegistry`, or (b) leave process-scaling as a stub that returns `success=false` (honest about the gateway's limitations) until an external supervisor integration is built. Option (b) is the lower-risk MVP and is what the snippet above implements.

## Caveats / Not Found

1. **No `buf.gen.yaml` checked in.** The TS proto regen is currently an undocumented manual step. The `buf-grpc-web-setup.md` research (archived) is the only place the config is written down. An implementer must create `packages/uc-orchestrator/buf.gen.yaml` (or run `protoc-gen-es` directly) and re-commit `engine_pb.ts`.

2. **Gateway cannot spawn worker processes.** `WorkerRegistry` (worker_service.rs:23-181) is purely an in-memory map of workers that self-register over gRPC. There is no process supervisor, no docker-compose control, and no NATS subject for "spawn a worker" in the gateway today (NATS subjects in server.rs:36-48 are task-submit, task-update, task-event, heartbeat, subtask-execute — none for worker lifecycle control). So `scale_to`/`scale_by` can only be fully implemented once an external supervisor integration is added; until then they should return `success=false` with an explanatory error (as the proposed Rust impl does).

3. **`desired_count` has no home.** Neither `WorkerRegistry` nor `GrpcServerInner` (server.rs:1094-1107) tracks a desired worker count. Implementing `scale_to` meaningfully requires adding state (e.g. `desired_count: AtomicU32` on `WorkerRegistry`).

4. **TS `WorkerService` client does not exist yet.** The TS `GrpcBridge` (grpc-bridge.ts) only wires `EngineService`, `TaskService`, `DashboardService`. Adding `scaleWorkers()` requires adding a `workerClient` field + instantiation in 3 spots (constructor, `reconnect()`, and the import list). The generated `WorkerService` descriptor is already present in `engine_pb.ts:2899`, so no codegen gap blocks this — only wiring.

5. **Python worker is unaffected.** The Python worker (nats_worker.py:1194, 1221) calls `register_worker_async`/`deregister_worker_async` via the Rust `GrpcEngineClient` (client.rs:187-255). A new `ScaleWorkers` RPC on `WorkerService` does not change the worker's registration flow. If Python ever needs to call `ScaleWorkers` (e.g. an admin CLI), `GrpcEngineClient` would gain an inherent `scale_workers()` method mirroring `register_worker` (client.rs:187) — but the task scope (TS `uc_worker` tool) does not require it.

6. **Streaming not needed.** `ScaleWorkers` is a unary RPC. The only streaming RPCs in the proto are `WatchTask` (TaskService) and `WatchDashboard` (DashboardService) + `SearchStream` (EngineService). Scaling is a one-shot command/response.
